# Service Desk Documentation Alignment (2026-05)

## Context

The Service Desk documentation began as high-level domain and architecture notes.
As implementation matured, the project became a more concrete, production-aligned portfolio demo.

By this point, the implementation direction was clearer in areas such as:

- LOCAL / REMOTE runtime separation
- server-side local demo mutable state
- route-handler orchestration boundaries
- AuthUser / SessionUser / AppUser separation
- session-aware impersonation
- derived ticket ownership
- action-oriented workflow rules
- explicit deferred scope for production-grade infrastructure

General docs and implementation direction started to drift.

---

## Problem

### 1. General docs lagged behind current implementation

Some documents still used older or purely conceptual wording.
This could cause misunderstandings around runtime, auth/session boundaries, ownership, and ticket operation behavior.

---

### 2. Historical logs and current docs have different responsibilities

The documentation set contains two different artifact types:

```txt
Current docs  -> explain the current system
Decision logs -> preserve historical reasoning at the time
```

If historical decision logs were rewritten to match the latest design,
historical context would be lost.

---

### 3. Portfolio credibility required explicit scope boundaries

The project is production-aligned, not production-complete.
Documentation needed to avoid overclaiming areas that are intentionally simplified or deferred
(for example full enterprise rule engine, full remote persistence, real-time infrastructure, or full compliance stack).

---

### 4. Domain documents needed stronger internal consistency

Service Desk behavior had to be described consistently as a workflow-oriented model:

- `Ticket`: workflow entity and current state
- `Activity`: meaningful user/operational interaction
- `History`: immutable event record generated by changes
- `Rules`: current executable behavior
- `Lifecycle`: valid state transition model

---

## Decision

Align non-decision-log markdown documentation with the current Service Desk design and implementation direction.

This keeps current documentation useful for reviewers while preserving decision logs as historical records.

### Scope Rules

- Do not modify existing historical decision logs.
- Update general design, architecture, strategy, and README documents to reflect the current design.
- Keep documentation changes minimal and focused.
- Avoid full rewrites unless necessary for consistency.
- Clarify implemented behavior, simplified demo behavior, and deferred production scope.
- Keep current strategy documents and operation rules aligned with each other.

---

## What Was Aligned

### 1. Runtime and Demo Strategy

- Clarified LOCAL / REMOTE runtime separation:

```txt
LOCAL  -> mock-backed demo behavior
REMOTE -> API / database-backed behavior
```

- Clarified orchestration boundary:

```txt
UI
-> feature API client
-> Next.js route handler
-> LOCAL handler or REMOTE proxy
```

- Clarified local demo mutable state as server-side in-memory state.
- Clarified reset behavior via `/api/demo/service-desk/reset`.

---

### 2. Local Demo Permission Scope

- Documented simplified local rule:

```txt
ADMIN     -> broad permission
non-ADMIN -> must be related to the ticket
```

- Kept full organization/department/job-field authorization as REMOTE expansion scope.

---

### 3. Auth / Session / App User Boundary

- Aligned around:

```txt
AuthUser    -> authentication identity
SessionUser -> session-safe projection
AppUser     -> application-facing user model
```

- Clarified:
  - `JWT` is authentication truth
  - session is auth projection
  - `SessionUser` excludes `accessToken`
  - Zustand store is runtime facade/cache, not auth truth

---

### 4. Session-Aware Impersonation

- Aligned impersonation as server/session-aware behavior (not client-only override).
- Clarified authorization boundary:
  - only `INTERNAL` users with at least `ADMIN` access can start impersonation
  - impersonation target must be a `TENANT` user
  - rule lives in auth layer, not UI

---

### 5. Derived Ownership

- Clarified ownership as derived, not fixed persisted fields.

```ts
type Ownership = {
  owner: boolean;
  assigned: boolean;
};
```

```txt
owner    = currentUser.username === ticket.requesterId
assigned = ticket.assigneeIds.includes(currentUser.username)
```

---

### 6. Activity / History / Lifecycle / Rules Consistency

- Reinforced workflow-oriented model over CRUD wording.
- Treated ticket operation rules as the implementation-facing source of truth for executable behavior.
- Aligned action model terminology and mutability rules.
- Clarified that communication actions (`comment`, `note`) can be edited/deleted under rules, while operational actions are immutable.
- Clarified closed-state edit/delete restrictions and soft delete (`active = false`).

---

### 7. Merge Semantics

- Standardized merge behavior:

```txt
source ticket -> Closed
closeReason = Merged
mergedIntoTicketId = target ticket id
```

- No separate `Merged` status.

---

### 8. UI Interaction Policy

- Aligned policy:

```txt
Page   -> primary workflow
Drawer -> secondary interaction
Dialog -> atomic action
```

- `ticket detail` as page (`/service-desk/[ticketId]`)
- `ticket creation` as dialog
- history/comment/action side panels as secondary interactions

---

### 9. Data Fetching and API Boundary

- Clarified React Query segmentation:
  - static/reference data
  - dynamic/mutable Service Desk data
- Clarified route handlers as orchestration boundary for LOCAL/REMOTE branching.

---

### 10. Feature and Module Boundary

- Clarified barrel export policy:
  - barrel files are public contracts
  - avoid dumping mixed server/client exports
- Clarified client-only shared utility boundary (for example `src/shared/client/`).

---

### 11. Link and Navigation Consistency

- Updated outdated README/docs links to current file structure.
- Adjusted wording to consistently represent current workflow-oriented and runtime-separated design.

---

## Consequences

### Positive

- Current docs now better reflect actual implementation direction.
- Domain and architecture terminology is more consistent.
- Runtime and auth/session boundaries are clearer to reviewers.
- Portfolio documentation is more credible about implemented vs deferred scope.

---

### Negative / Trade-offs

- Ongoing documentation maintenance requires stricter discipline.
- Terminology precision increases editing cost.
- Current docs and decision logs must continue to be maintained with distinct purposes.

---

## Follow-up Policy

- General documentation should describe the current system.
- Decision logs should preserve the context and rationale of major changes.
- If a documentation alignment creates a new architectural meaning, it should be captured in a decision log.
- Deferred production concerns should remain explicitly marked as future scope.
- Service Desk docs should continue to distinguish domain model, implementation strategy, and executable rules.

---

## Summary

We aligned non-decision-log Service Desk documentation with the current architecture and implementation strategy while preserving historical decision logs as-is.

This improves clarity, consistency, and credibility of the portfolio documentation without rewriting historical records.
