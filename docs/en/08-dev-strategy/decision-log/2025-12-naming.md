# Naming Decision (2025-12)

## Context

Designing the Service Desk project required a consistent naming strategy across:

- UI components
- form modes
- i18n keys
- component APIs

Several naming ambiguities appeared during implementation:

- `Settings` vs `Preferences`
- `create` vs `edit` vs `view` mode
- field translation key structure
- helper function abstraction vs explicit key usage
- component APIs such as `children` vs `trigger`

At first, these looked like simple naming issues.
Over time, it became clear that naming was not just cosmetic, but part of the system design itself.

---

## Problem

### 1. Semantic Ambiguity

#### Example: `Settings`

The word `Settings` is usually understood as:

- system-level configuration
- admin configuration

However, the actual features being implemented were:

- language selection
- theme selection (light/dark)
- color theme selection

This is closer to **user preferences** than system settings.

---

### 2. Mode Naming Ambiguity

The initial form prop looked like this:

```ts
type?: "create" | "edit";
```

Problems:

- the UI may look similar, but the lifecycle is different
- `edit` is semantically vague
- it does not align clearly with API intent (`POST` vs `PUT`)

---

### 3. i18n Naming and Access

Two approaches were considered:

```ts
t(`field.${name}.label`);
```

vs

```ts
fieldLabel(name);
```

Problems:

- abstraction vs explicitness
- code readability during implementation
- the need for a structure understandable even to non-developers such as HR or reviewers

---

### 4. Component API Naming

For dialog trigger design:

```tsx
<Dialog trigger={...} />
```

vs

```tsx
<Dialog>{trigger}</Dialog>
```

Problems:

- flexibility vs explicitness
- API consistency

---

## Decision Summary

```txt
Naming should be defined by meaning, not just implementation shape.
```

---

## 1. `Settings` vs `Preferences`

### Decision

```txt
User-facing personalization is named "Preferences", not "Settings".
```

### Rationale

- `Settings` implies system or admin configuration
- `Preferences` better represents user personalization

### Result

```tsx
<PreferencesMenu />
```

This keeps the user-facing UX semantics clear.

---

## 2. Form Mode Naming

### Decision

```ts
type Mode = "create" | "update" | "view";
```

### Rationale

- `edit` is ambiguous
- `update` matches API semantics more clearly
- even when the UI is similar, `create` and `update` represent different lifecycle states

---

### Key Insight

```txt
If the lifecycle is different, the name should also be different.
```

---

## 3. Explicit i18n Key Strategy

### Decision

```txt
Prefer explicit i18n keys over helper abstraction.
```

### Example

```ts
t("field.title.label", { ns: "common" });
```

### Rationale

- The meaning is visible directly in the code
- It is easier for non-developers or reviewers to understand
- It avoids hidden logic behind helper abstractions

---

### Rejected Option

```ts
fieldLabel("title");
```

Problems:

- the namespace is not obvious
- tracing behavior becomes harder because the meaning is hidden behind abstraction

---

## 4. Field Naming Structure

### Decision

```json
{
  "field": {
    "title": {
      "label": "...",
      "placeholder": "..."
    }
  }
}
```

### Rationale

- Keeps `label` and `placeholder` organized together
- Improves reuse
- Maintains structural consistency

---

## 5. Component API Naming

### Decision

```txt
Use a `trigger` prop by default, but allow a children-based pattern when needed.
```

### Example

```tsx
<Dialog trigger={<Button />} />
```

or

```tsx
<Dialog>{trigger}</Dialog>
```

### Rationale

- Supports explicit APIs
- Preserves flexibility when composition is more natural

---

### Insight

```txt
Component APIs should reveal how the component is meant to be used.
```

---

## 6. Naming Principles

### Rule 1

```txt
Names should express meaning and responsibility, not only implementation form.
```

---

### Rule 2

```txt
If the UI looks similar but the lifecycle is different, use different names.
```

---

### Rule 3

```txt
Prefer explicitness over unnecessary abstraction.
```

---

### Rule 4

```txt
Use naming that remains understandable to non-developers as well.
```

---

## Trade-offs

### Pros

- Enables a meaning-driven design approach
- Improves code readability
- Makes collaboration easier
- Reinforces domain understanding

---

### Cons

- Naming can become slightly more verbose
- It requires more deliberate design decisions early on

---

## Impact

### Code Quality

- Naming alone becomes enough to understand the structure more quickly

---

### Documentation

- Code and documents can share the same semantics

---

### Collaboration

- Easier for HR, planners, designers, and reviewers to understand

---

## Final Insight

```txt
Naming is not an implementation detail.
It is the smallest visible unit that reveals system design.
```

This decision is not just about cleaning up names.
It is a choice to reflect the domain-driven structure consistently all the way down to code-level vocabulary.
