# Ticket History

## Goal

The ticket history model is designed to track all meaningful changes and events
that occur throughout the lifecycle of a ticket.

It ensures:

- full traceability of actions
- accountability of users
- auditability for compliance
- transparency in operations

---

## Core Concept

Every significant ticket change generates a history event.

```txt
Action -> Event -> Stored History Record
```

The system is event-driven in terms of tracking, not state-driven.
History is always generated as a result of activity execution,
ensuring that all changes are traceable and immutable.

---

## Relationship with Ticket Activity

Ticket history and ticket activity serve related but different purposes.

### Ticket Activity

- represents user-facing communication and operational actions
- stores intent, message, and structured action context
- powers the unified ticket timeline

### Ticket History

- records immutable event traces generated by ticket changes
- emphasizes before/after values and auditability
- preserves a lower-level operational record of what occurred

### Practical Distinction

```txt
Activity = meaningful interaction or operation
History = immutable event record generated by change
```

This distinction helps the domain stay expressive at the UI level while
remaining precise for audit and debugging.

---

## What is Tracked

The system records all critical ticket changes.

### 1. Status Changes

- created
- approved
- assigned
- working
- pending
- resolved
- closed

### 2. Field Changes

- priority
- risk level
- category
- assignee
- description, when needed

### 3. Assignment Events

- assigned
- reassigned
- unassigned

### 4. SLA Events

- SLA started
- SLA paused
- SLA resumed
- SLA breached

### 5. Approval Events

- approval requested
- approved
- rejected

### 6. System Events

- auto-assignment triggered
- escalation triggered
- fallback assignment applied

## History Record Structure

Each history entry follows a consistent structure.

```ts
type History = {
  id: string;
  ticketId: string;

  type: HistoryType;
  action: string;

  actorId: string | null; // null for system actions

  fromValue?: unknown;
  toValue?: unknown;

  metadata?: Record<string, unknown>;

  createdAt: Date;
};
```

### Example Enum

type HistoryType =

## Actor Model

Each history record identifies who performed the action.

### Actor Types

- user
- requester
- system

---

## Event Granularity

### Fine-Grained Tracking

Each change is recorded as an individual event.

#### Example

```txt
Priority: Medium -> High
Status: Open -> Working
```

This creates two separate history records.

### Trade-Off

| Approach     | Pros               | Cons            |
| ------------ | ------------------ | --------------- |
| Fine-grained | Precise, auditable | More records    |
| Aggregated   | Less storage       | Harder to trace |

---

## Read Model vs Write Model

### Write Model

- optimized for storing events
- append-only structure

### Read Model

- optimized for timeline display
- may aggregate or format entries for UI readability

---

## Immutability

History records are immutable.

### Rules

- cannot be updated
- cannot be deleted in normal operations
- corrections are recorded as new events

---

## Ordering and Consistency

History must be chronologically ordered.

### Considerations

- use server timestamps
- avoid client-side ordering
- handle concurrent updates carefully

---

## UI Representation

### Timeline View

- chronological list of events
- most recent first, or toggleable ordering

### Example

```txt
[10:32] Assigned to John
[10:30] Status changed to Working
[10:20] Ticket Created
```

### Key UX Elements

- clear action description
- visible actor
- timestamp
- emphasis for important changes

---

## Metadata Usage

Metadata provides additional context when plain before/after values are not
enough.

### Example

```ts
metadata: {
  reason: "Manual reassignment",
  previousAssignee: "user_1",
}
```

---

## System vs User Actions

System-generated events must be distinguishable from user actions.

### Examples

- "Auto-assigned by system"
- "Escalated due to SLA breach"

---

## Performance Considerations

### Challenges

- high volume of history records
- frequent writes

### Solutions

- indexed queries by `ticketId` and `createdAt`
- pagination in UI
- archiving older records when necessary

---

## Integration with Other Domains

### SLA

- logs SLA lifecycle events

### Assignment

- tracks ownership changes

### Approval

- records approval decisions

### Track Time

- records start, finish, switch, and manual corrections

---

## Edge Cases

### 1. Rapid Consecutive Updates

- multiple events may occur within milliseconds
- ordering still needs to remain deterministic

### 2. Bulk Updates

- one action may change multiple fields
- the system must decide whether to split or aggregate those changes

### 3. System Failures

- partial history writes are not acceptable
- transactional handling is preferred where possible

### 4. Sensitive Data Changes

- sensitive values should not be stored blindly
- masking or omission may be required

---

## Why Not Full Event Sourcing?

This system adopts a hybrid approach:

- state-based main model
- event-based history tracking

### Reason

- simpler implementation
- sufficient audit capability
- better performance balance
