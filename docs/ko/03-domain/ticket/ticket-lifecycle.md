# 티켓 생명주기

## 목적

티켓 생명주기는 티켓이 생성부터 완료까지 어떻게 진행되는지를 정의한다.

이를 통해 다음을 보장한다.

- 모든 티켓이 예측 가능한 워크플로를 따른다.
- 각 단계에서 책임이 명확하게 정의된다.
- 상태 전이가 통제되고 추적 가능하다.
- 시스템이 실제 운영 프로세스를 반영한다.

---

## 핵심 개념

```txt
A ticket is a stateful workflow, not a static record.
```

티켓은 다음 요소에 따라 정의된 상태를 거쳐 이동한다.

- 사용자 액션
- 승인 결정
- 시스템 규칙

모든 티켓이 모든 단계를 반드시 거치는 것은 아니다.
실제 경로는 카테고리 설정에 따라 달라진다.

---

## 생명주기 흐름

상위 수준에서 생명주기는 다음처럼 요약할 수 있다.

```txt
Draft -> Open -> Approval -> Working -> Resolved -> Closed
```

이것은 단순화된 개요이며, 실제로는 다음과 같은 경우가 있다.

- 어떤 티켓은 `Open` 에서 바로 `Working` 으로 이동한다.
- 어떤 티켓은 실행 전에 승인 단계를 거친다.
- 어떤 티켓은 승인 거절로 조기 종료된다.

---

## 상태 정의

### Draft

- 제출 전 임시 상태
- 요청자만 수정 가능
- 담당자에게는 보이지 않음
- 제출 전 미완성 요청을 준비하기 위해 사용됨

### Open

- 티켓이 제출된 상태
- 생성 후 최초의 활성 상태
- 승인 또는 할당을 기다리는 상태

### Approval

- 티켓이 승인 프로세스에 들어간 상태
- 하나 이상의 승인 단계가 존재할 수 있음
- 진행 여부는 승인 결정에 따라 달라짐

### Working

- 티켓이 할당되어 처리 중인 상태
- 담당자가 실행 책임을 가짐
- 작업 세션이 기록될 수 있음

### Resolved

- 작업이 완료된 상태
- 해결안이 제공된 상태
- 확인 또는 종료를 기다리는 상태

### Closed

- 최종 상태
- 일반적으로 더 이상의 수정은 기대되지 않음
- 티켓이 완료된 것으로 간주됨

---

## 상태 전이

생명주기는 명시적인 상태 전이에 의해 진행된다.

### 1. Draft -> Open

트리거:

- 사용자 제출

### 2. Open -> Approval

조건:

- 카테고리에 승인 요구사항이 있음

### 3. Open -> Working

조건:

- 승인이 필요하지 않음

### 4. Approval -> Working

트리거:

- 승인 완료

### 5. Approval -> Closed

트리거:

- 승인 거절

### 6. Working -> Resolved

트리거:

- 담당자가 작업을 완료함

### 7. Resolved -> Closed

트리거:

- 요청자 확인
- 시스템 자동 종료

---

## 조건부 워크플로

생명주기는 모든 티켓에 대해 고정되어 있지 않다.

생명주기는 카테고리 설정에 따라 달라진다.

```txt
Category -> determines -> Workflow
```

예시:

- 단순 요청: `Open -> Working -> Closed`
- 승인이 필요한 요청: `Open -> Approval -> Working -> Closed`

---

## 전이 규칙

### 1. 통제된 전이

상태 전이는 임의로 일어나지 않는다.

반드시 다음을 따라야 한다.

- 승인 결과
- 할당 상태
- 시스템 제약

### 2. 직접 점프 금지

유효하지 않은 전이는 허용되지 않는다.

예시:

- `Draft -> Working`
- `Open -> Resolved`
- 승인 결정 없이 `Approval -> Closed`

### 3. 상태 일관성

각 상태는 명확한 책임을 표현해야 한다.

- `Open`: 시스템 결정을 기다리는 상태
- `Approval`: 승인자를 기다리는 상태
- `Working`: 담당자가 소유하는 상태
- `Resolved`: 종료 결정을 기다리는 상태

---

## 생명주기와 상태의 차이

생명주기는 전체 흐름을 설명하고, 상태는 현재 시점을 나타낸다.

```txt
Lifecycle = process
Status = snapshot
```

이 구분은 중요하다. 티켓 도메인은 단순히 현재 상태만 저장하는 것이 아니라,
현재 상태에 도달한 과정을 함께 모델링하기 때문이다.

---

## 다른 도메인과의 관계

### Category

- 승인이 필요한지 결정한다.
- 할당 규칙을 정의한다.
- 어떤 워크플로 경로가 적용되는지 결정한다.

관련 문서: [Category Strategy](./strategy/category-strategy.md)

---

### Approval

- 승인 단계에서의 전이를 제어한다.
- 티켓이 계속 진행될지 중단될지를 결정한다.

관련 문서: [Approval System](./strategy/approval-system.md)

---

### Assignment

- `Working` 상태에서 누가 티켓을 처리하는지 결정한다.
- 운영 소유권에 영향을 준다.

관련 문서: [Assignment Policy](./strategy/assignment-policy.md)

---

### SLA

- 긴급도와 마감 시간을 좌우한다.
- 우선순위화와 모니터링에도 영향을 줄 수 있다.

관련 문서: [SLA Strategy](./strategy/sla-strategy.md)

---

### Ticket History

- 모든 상태 전이를 기록한다.
- 생명주기 변경 사항을 감사 가능하고 명시적으로 만든다.

상태 변화는 암묵적으로 일어나지 않는다.
모든 전이는 워크플로 이벤트로 기록된다.

관련 문서: [Ticket History](./ticket-history.md)

---

## 설계 고려사항

### 1. 카테고리를 통한 유연성

생명주기는 티켓 유형별로 하드코딩되지 않고, 의도적으로 구성 가능하게 설계되었다.

### 2. 현실적인 운영 흐름 정렬

생명주기는 실제 운영 흐름을 반영한다.

- 실행 전 검증
- 책임 이관
- 완료 확인

### 3. 추적 가능성

모든 전이는 기록되며, 이를 통해 감사 가능성과 디버깅 가능성을 확보한다.

---

## 요약

티켓 생명주기는 티켓이 통제된 워크플로를 따라 이동하도록 보장한다.

- 티켓은 구조화된 흐름을 따라 움직이고
- 각 단계에서 책임은 명확하며
- 전이는 예측 가능하고 감사 가능하다.

이 생명주기는 신뢰할 수 있고 확장 가능한 서비스 데스크 시스템을 만드는 핵심 기반이다.
