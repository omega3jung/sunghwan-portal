# 티켓 시스템 개요

## 목적

티켓 시스템은 요청, 승인, 실행 워크플로를 구조적이고 추적 가능한 방식으로
관리하기 위해 설계되었다.

단순한 CRUD 기반 시스템과 달리, 이 시스템은 다음에 초점을 둔다.

- 일관된 워크플로 실행
- 명확한 소유권과 책임
- 모든 액션에 대한 완전한 감사 가능성
- 현실적인 작업 시간 추적
- 사용자 중심 우선순위화

이 시스템의 목표는 단순히 데이터를 저장하고 보여주는 것이 아니라,
실제 운영 프로세스를 반영하는 것이다.

---

## 왜 단순한 CRUD 시스템이 아닌가?

많은 티켓 시스템은 다음과 같은 기본 흐름을 따른다.

```txt
Create -> Update -> Resolve -> Close
```

하지만 실제 운영 환경에서는 이 접근만으로는 충분하지 않다.

- 요청 유형마다 서로 다른 워크플로가 필요하다.
- 실행 전에 승인이 필요할 수 있다.
- 책임 주체가 명확하게 할당되어야 한다.
- SLA가 강제되어야 한다.
- 작업은 자주 중단되었다가 다시 이어진다.

그 결과, CRUD 기반 시스템은 다음과 같은 문제를 만든다.

- 일관성 없는 동작
- 불명확한 책임
- 실제 작업에 대한 낮은 가시성

이 문제를 해결하기 위해 이 시스템은 티켓을 워크플로 중심 엔티티로 모델링한다.

---

## 핵심 원칙

이 시스템은 다음 원칙을 기반으로 설계되었다.

### 1. 카테고리 중심 동작

카테고리는 단순 분류값이 아니다.

카테고리는 다음을 정의한다.

- 할당 규칙
- 승인 요구사항
- SLA 정책
- 워크플로 구조

### 2. 승인 워크플로의 일급 시민화

승인은 생명주기의 핵심 요소로 다뤄진다.

- 카테고리별로 구성 가능하다.
- 구조화된 파이프라인으로 실행된다.
- 전체 과정이 추적 가능하다.

### 3. 세션 기반 작업 추적

작업은 하나의 누적 값으로 저장되지 않는다.

```txt
work = collection of sessions
```

이를 통해 다음이 가능해진다.

- 작업 중단
- 작업 재개
- 작업 전환

### 4. 작업 컨텍스트 기반 우선순위화

티켓은 단순한 평면 목록으로 표시되지 않는다.

시스템은 작업 컨텍스트를 계산하여 다음을 우선시한다.

- 현재 진행 중인 작업
- 나에게 할당된 티켓
- 즉시 처리 가능한 티켓
- 문맥상 중요한 티켓

### 5. 완전한 감사 가능성

중요한 모든 액션은 기록된다.

이를 통해 다음이 가능해진다.

- 추적 가능성
- 책임 추적
- 설명 가능성

---

## 시스템 구조

티켓 시스템은 여러 도메인 구성 요소로 이루어진다.

```txt
Ticket
  -> Category
  -> Approval
  -> Assignment
  -> SLA
  -> Track Time
  -> History
```

각 구성 요소는 특정 책임만 담당하며,
이를 통해 시스템은 모듈형 구조와 유지보수성을 확보한다.

---

## 티켓 생명주기

티켓은 구조화된 생명주기를 따라 진행된다.

```txt
Draft -> Open -> Approval -> Working -> Resolved -> Closed
```

실제 흐름은 카테고리 설정에 따라 달라진다.

- 어떤 티켓은 승인을 생략한다.
- 어떤 티켓은 여러 승인 단계를 요구한다.

관련 문서: [Ticket Lifecycle](./ticket-lifecycle.md)

---

## 작업 세션 모델

티켓에서 수행된 작업은 세션 단위로 추적된다.

```txt
start -> work -> finish
```

하나의 티켓에는 여러 세션이 존재할 수 있다.

예시:

```txt
09:00 - 10:00
15:00 - 17:00
```

이 방식은 하나의 누적 시간 값보다 실제 작업 흐름을 더 정확하게 반영한다.

관련 문서: [Ticket Track Time](./ticket-track-time.md)

---

## 작업 컨텍스트

시스템은 사용자별 현재 작업 컨텍스트를 계산한다.

이를 통해 다음이 가능해진다.

- 우선순위가 적용된 티켓 목록
- 컨텍스트 인지형 UI 액션
- 인지 부하 감소

예시:

```txt
working on a ticket -> show Finish
working on another ticket -> show Switch
```

관련 문서: [Ticket Track Time](./ticket-track-time.md)

---

## 설계 철학

이 시스템은 다음을 우선한다.

- 유연성보다 예측 가능성
- 하드코딩보다 설정 가능성
- 파괴적 단순성보다 추적 가능성
- UI 편의보다 운영 현실성

이러한 선택은 시스템이:

- 복잡성 증가에도 확장 가능하고
- 일관성을 유지하며
- 이해하기 쉬운 구조를 갖도록 만든다.

## 요약

이 티켓 시스템은 워크플로 중심이면서 사용자 중심인 시스템으로 설계되었다.

핵심 특징은 다음과 같다.

- 카테고리 중심 동작
- 승인 인식형 흐름
- 세션 기반 작업 추적
- 컨텍스트 기반 우선순위화
- 완전한 감사 가능성

이 시스템은 현실적이고 확장 가능한 서비스 데스크를 구축하기 위한 기반을 제공한다.
