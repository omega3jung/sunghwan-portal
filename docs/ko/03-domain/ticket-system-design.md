# 티켓 시스템 설계

## 목표

티켓 시스템은 조직 전반의 사용자 요청을 처리하기 위해
**구조화되고, 추적 가능하며, 확장 가능한 워크플로우**를 제공하도록 설계된다.

이 시스템이 중점적으로 다루는 요소는 다음과 같다.

- 일관된 요청 처리
- 명확한 소유권과 책임
- 완전한 감사 가능성
- SLA 기반 운영 추적

---

## 핵심 원칙

이 시스템은 다음 원칙을 기반으로 구성된다.

1. **카테고리 주도 워크플로우**
2. **설정 가능한 승인 파이프라인**
3. **파괴적 삭제 금지**
4. **완전한 감사 추적**
5. **SLA 기반 우선순위화**

이 원칙들은 프런트엔드와 백엔드 전반에서 일관되게 유지되어야 한다.

---

## 시스템 개요

티켓 시스템은 다음 핵심 구성요소로 이루어진다.

- Ticket
- Category (Main / Sub)
- Approval System
- Assignment System
- SLA Tracking
- History (Audit Trail)
- Time Tracking

---

## 티켓 생명주기

티켓은 구조화된 생명주기를 따른다.

```id="x8g1px"
Pre -> Open -> Approved / Declined -> Working -> Pending -> Resolved -> Closed
```

---

## 상태 정의

### Pre

- 티켓이 생성된 상태
- 승인 프로세스가 아직 시작되지 않은 상태

---

### Open

- 승인 프로세스가 진행 중인 상태

---

### Approved

- 모든 승인 단계가 성공적으로 완료된 상태
- 티켓이 할당 가능한 상태가 된 상태

---

### Declined

- 어느 한 승인 단계에서라도 승인에 실패한 상태
- 티켓이 반려된 상태

---

### Working

- 담당자가 티켓 처리를 시작한 상태
- 티켓이 **잠금(locked)** 상태가 된다

---

### Pending

- 요청자 입력 또는 추가 데이터를 기다리는 상태

---

### Resolved

- 작업이 완료된 상태
- 요청자 확인을 기다리는 상태

---

### Closed

- 티켓 생명주기가 완료된 상태

---

## 티켓 잠금 정책

티켓이 **Working** 상태에 진입하면:

- 핵심 정보는 변경 불가능해진다
- 수정 작업이 제한된다

### UI 동작

- `InfoStep`은 숨겨진다
- `ReviewStep`만 사용할 수 있다

### 목적

- 실행 중 비일관적인 변경을 방지한다
- 운영 무결성을 유지한다

---

## 카테고리 연계

모든 티켓은 반드시 하나의 카테고리와 연결되어야 한다.

### 카테고리의 책임

- 기본 우선순위와 위험 수준을 정의한다
- SLA를 결정한다
- 승인 요구사항을 정의한다
- 할당 로직을 구동한다

참고: `category-strategy.md`

---

## 승인 연계

일부 티켓은 실행 전에 승인이 필요하다.

### 동작 방식

- 티켓 생성 후 승인 프로세스가 시작된다
- 카테고리 설정에 의해 제어된다
- 승인 단계는 순차적으로 진행된다

참고: `approval-system.md`

---

## 할당 정책

티켓은 기본적으로 카테고리 설정에 따라 할당된다.

### 흐름

```id="g8a0p7"
Ticket Approved -> Assignment Resolved -> Assigned -> Working
```

### 특성

- 기본값은 자동 할당이다
- 수동 재할당도 허용된다

---

## 소유권 모델

각 티켓은 다음과 같은 소유권 플래그를 노출한다.

```ts id="xv7zq4"
type Ownership = {
  owner: boolean;
  assigned: boolean;
};
```

### 정의

- `owner`: 현재 사용자가 요청자이다
- `assigned`: 현재 사용자가 담당자 중 한 명이다

### 목적

- 권한 처리를 단순화한다
- UI 조건부 렌더링을 개선한다

---

## SLA 추적

SLA는 카테고리 설정을 기반으로 계산된다.

### 규칙

```id="5y5y0v"
dueDate = createdDate + defaultSlaDays
```

### 동작

- SLA는 우선순위와 긴급도에 영향을 준다
- 모니터링과 리포팅에 활용된다

---

## 작업 시간 추적

작업 시간은 담당자별로 추적된다.

### 구조

```ts id="1n9w3n"
type TrackTime = {
  ticketId: string;
  assigneeId: string;
  time: number;
};
```

### 목적

- 작업 부하를 측정한다
- SLA 분석을 지원한다
- 성과 인사이트를 도출할 수 있게 한다

---

## 이력 (감사 추적)

모든 변경 사항은 `History` 모델에 기록된다.

### 추적 대상 이벤트

- 승인 액션
- 상태 변경
- 담당자 할당 변경
- 노트 및 코멘트

### 구조

```ts id="9x3kpf"
type History = {
  ticketId: string;
  type: string;
  userId: string;
  date: Date;
  note?: string;
  active: boolean;
};
```

### 목적

- 완전한 추적 가능성 확보
- 디버깅과 감사 지원
- 컴플라이언스 요구사항 충족

---

## 삭제 정책

티켓은 영구 삭제하지 않는다.

대신 다음과 같이 비활성화한다.

```id="8b7txj"
active = false
```

### 목적

- 감사 이력을 보존한다
- 리포팅 무결성을 유지한다
- 추적 가능성을 보장한다

---

## 첨부파일

티켓은 파일 및 이미지 첨부를 지원한다.

### 구조

```ts id="ttkq3n"
type Attachment = {
  index: number;
  type: "file" | "image";
  name: string;
  url: string;
  active: boolean;
};
```

### 동작

- 비활성 첨부파일은 삭제하지 않고 숨긴다

---

## 엣지 케이스

### 1. 승인 없는 티켓

- 일부 카테고리는 승인 단계를 건너뛴다
- 티켓은 바로 할당 단계로 이동한다

---

### 2. 재할당

- 티켓은 `Working` 상태 중에도 재할당될 수 있다
- 이 변경은 반드시 이력에 기록되어야 한다

---

### 3. Pending 상태

- 티켓은 `Working` 과 `Pending` 사이를 오갈 수 있다
- UI에서 이를 명확하게 표시해야 한다

---

### 4. SLA 위반

- `dueDate`를 초과하면 티켓에 플래그가 표시되어야 한다 (UI/모니터링)

---

## 트레이드오프

### 장점

- 매우 구조화된 워크플로우
- 강력한 감사 가능성
- 조직 전반으로 확장 가능
- 관심사의 명확한 분리

---

### 단점

- 시스템 복잡도가 증가한다
- 엄격한 데이터 거버넌스가 필요하다
- 초기 설계 비용이 더 많이 든다

---

## 고려한 대안

### 1. 단순 CRUD 티켓 시스템

- 구현이 쉽다
- 워크플로우 제어와 추적 가능성이 부족하다

---

### 2. 티켓별 완전 유연 워크플로우

- 매우 높은 커스터마이징이 가능하다
- 일관성이 떨어지고 유지보수가 어렵다

---

### 3. 감사 추적 없음

- 저장 모델이 더 단순하다
- 엔터프라이즈 환경에는 적합하지 않다

---

## 설계 철학

이 시스템은 다음을 우선한다.

- 유연성보다 예측 가능성
- 하드코딩보다 설정 기반 설계
- 단순성보다 추적 가능성

---

## 요약

티켓 시스템은 **견고하고 엔터프라이즈 환경에 적합한 워크플로우 엔진**을 제공하며,
카테고리 기반 설정과 엄격한 생명주기 관리를 통해
일관성, 책임성, 확장성을 보장한다.
