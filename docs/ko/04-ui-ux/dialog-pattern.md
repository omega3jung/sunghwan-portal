# Dialog 패턴

## 목표

Dialog 패턴은 통제된 UI 환경에서 사용자 입력과 액션을 처리하기 위한
**일관되고 확장 가능한 상호작용 모델**을 제공하도록 설계되었다.

이 패턴의 목표는 다음과 같다.

- 시스템 전반에서 dialog 사용 방식을 표준화한다
- 일시적인 상호작용과 주요 워크플로를 분리한다
- 사용자 경험의 일관성을 높인다
- UI 복잡도를 줄인다

---

## 핵심 원칙

```txt
Dialog는 주요 워크플로가 아니라 일시적인 상호작용에 사용한다
```

---

## Dialog와 Page

### 규칙

| Dialog 사용 | Page 사용 |
| ---------- | --------- |
| 빠른 액션 | 핵심 워크플로 |
| 짧은 폼 | 복잡한 폼 |
| 임시 상호작용 | 오래 지속되는 상호작용 |

---

### 예시

- 티켓 생성 -> Dialog
- 티켓 수정(복잡한 경우) -> Page
- 티켓 상세 보기 -> Page

---

## Dialog 유형

### 1. Form Dialog

데이터 생성 또는 수정을 위해 사용한다.

#### 예시

- 티켓 생성
- 빠른 수정

---

### 2. Confirmation Dialog

파괴적이거나 중요한 액션에 사용한다.

#### 예시

- 티켓 삭제
- 티켓 종료

---

### 3. Informational Dialog

수정할 수 없는 정보를 보여주기 위해 사용한다.

---

## TicketFormDialog 패턴

시스템은 티켓 생성에 **단계형 dialog 패턴**을 사용한다.

---

### 구조

```tsx
<Dialog>
  <DialogTrigger />
  <DialogContent>
    <Header />
    <StepContent />
    <Footer />
  </DialogContent>
</Dialog>
```

---

### 특징

- 다단계 흐름
- 제어된 이동(`next` / `prev`)
- 마지막 단계에서 최종 제출

---

## 상태 관리

### 제어형 상태

Dialog의 open 상태는 **외부에서 제어**한다.

```tsx
const [open, setOpen] = useState(false);
```

---

### 왜 제어형인가?

- mutation 이후 dialog를 닫을 수 있다
- 프로그래밍 방식으로 제어할 수 있다
- 비즈니스 로직과의 통합이 쉬워진다

---

## Trigger 전략

### 규칙

```txt
DialogTrigger는 선택 가능해야 하며 교체 가능해야 한다
```

---

### 패턴

- `trigger`를 prop으로 받는다
- 없으면 기본 버튼으로 대체한다

---

### 예시

```tsx
<Dialog trigger={trigger ?? <DefaultButton />} />
```

---

## 닫기 전략

### 방법

1. 사용자 상호작용(닫기 버튼)
2. 바깥 영역 클릭
3. 성공 이후 프로그래밍 방식으로 닫기

---

### 권장 방식

```tsx
onSuccess: () => setOpen(false);
```

---

## Stepper 패턴

### 언제 사용하는가?

- 복잡한 폼
- 여러 의존 입력이 있는 경우

---

### 현재 결정

- 현재는 `TicketFormDialog`에만 구현한다
- 아직 전역 공통 컴포넌트로 추상화하지 않는다

---

### 이유

- 너무 이른 추상화를 피한다
- 향후 변경에 대한 유연성을 유지한다

---

## 데이터 흐름

### 규칙

```txt
필요하지 않다면 Dialog 안에서 데이터를 가져오지 않는다
```

---

### 권장 방식

- 필요한 데이터는 props로 전달한다
- dialog 자체가 진입점일 때만 fetch한다

---

## 검증 전략

- dialog 내부에서 `react-hook-form`을 사용한다
- 단계별 또는 제출 시점에 검증한다

---

## UX 고려 사항

### 1. 포커스 관리

- dialog가 열리면 포커스가 dialog 내부로 이동해야 한다
- dialog가 닫히면 포커스가 trigger로 돌아가야 한다

---

### 2. 스크롤 처리

- 내용이 길다면 scroll area를 사용한다

---

### 3. Escape 처리

- 별도 제한이 없으면 `ESC`로 dialog를 닫는다

---

### 4. 접근성

- 적절한 ARIA role을 사용한다
- 키보드 내비게이션을 지원한다

---

## 성능 고려 사항

### 지연 렌더링

- dialog가 열릴 때만 내용을 렌더링한다

---

### 무거운 초기화 피하기

- 비용이 큰 로직은 dialog가 열린 뒤로 미룬다

---

## 피하려는 안티 패턴

### 1. 중첩 Dialog

- dialog 안에 또 다른 dialog를 넣는 구조

---

### 2. 주요 워크플로를 Dialog에 넣는 것

- page에 가까운 복잡한 워크플로를 dialog 안에 넣는 구조

---

### 3. 비제어형 Dialog 상태

- open/close 동작을 신뢰성 있게 제어할 수 없는 구조

---

### 4. 깊은 UI 계층에서의 데이터 가져오기

- dialog 하위 컴포넌트 안에서 API 호출을 수행하는 구조

---

## 트레이드오프

### 장점

- 일관된 상호작용 패턴을 제공한다
- 빠른 액션에 더 좋은 UX를 제공한다
- 페이지 이동 부담을 줄인다

---

### 단점

- 복잡한 UI를 담기에는 공간이 제한된다
- 상태 관리가 필요하다
- 과도하게 사용하면 구조가 복잡해질 수 있다

---

## 고려한 대안

### 1. Page 기반만 사용하는 방식

- 장점: 라우팅이 단순하다
- 단점: 빠른 액션에는 상호작용이 느려진다

---

### 2. Drawer 기반만 사용하는 방식

- 장점: 레이아웃이 유연하다
- 단점: 동작을 표준화하기 어렵다

---

### 3. 전역 Dialog Manager

- 장점: 중앙집중식 제어가 가능하다
- 단점: 복잡도가 증가한다
- 단점: 디버깅이 더 어려워진다

---

## 설계 원칙과의 정렬

이 패턴은 다음 원칙과 맞닿아 있다.

- 관심사의 분리
- UX 일관성
- 제어된 상태 관리
- 확장 가능한 UI 아키텍처

---

## 요약

Dialog 패턴은 일시적인 상호작용을 처리하기 위한
**구조적이고 일관된 접근 방식**을 제공한다.

이를 통해 dialog는 가볍고 예측 가능하며,
전체 시스템 아키텍처와 정렬된 상태로 유지될 수 있다.
