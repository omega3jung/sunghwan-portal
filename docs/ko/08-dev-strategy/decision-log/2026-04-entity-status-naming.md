# 엔티티 상태 네이밍 결정 (2026-04)

## Context

`Category`, `Ticket`, `Attachment`와 같은 도메인 엔티티를 설계하면서,
엔티티가 현재 진행 중인 작업에서 계속 사용 가능한지를 일관되게 표현할 방법이 필요했다.

처음에 검토한 네이밍 후보는 다음과 같았다.

- `active` / `inactive`
- `enabled` / `disabled`
- 전용 `disabled` boolean 플래그

처음에는 이 선택지들이 서로 바꿔 써도 될 것처럼 보였다.
하지만 구현을 진행하면서, 이들은 서로 다른 의미 계층을 나타낸다는 점이 분명해졌다.

---

## Problem

### 1. 도메인 상태와 UI 상태

서로 다른 두 개념이 혼용되고 있었다.

- **도메인 상태**
  - 해당 엔티티가 새로운 비즈니스 작업에 유효한지 여부
- **UI 상태**
  - 현재 컴포넌트가 상호작용 가능한지 여부

이 두 개념은 관련은 있지만 동일하지는 않다.

예를 들어:

- 어떤 카테고리는 더 이상 새 티켓에서 선택할 수 없을 수 있다
- 어떤 셀렉트 입력은 권한, 로딩 상태, 혹은 파생된 비즈니스 규칙 때문에 비활성화될 수 있다

두 개념에 같은 단어를 쓰면 모델이 덜 명확해진다.

---

### 2. 도메인 레벨에서의 `disabled` 모호성

도메인 모델에서 `disabled`를 사용하면 의미가 불분명해진다.

- 비즈니스 정책에 의해 비활성화된 것인가?
- 일시적으로 사용할 수 없는 것인가?
- 단지 UI를 위한 플래그인가?

`disabled`는 HTML과 컴포넌트 동작과 강하게 연결되어 있기 때문에,
도메인 모델에 UI 의미를 끌어들이게 된다.

---

## Decision Summary

```txt
도메인 엔티티는 `active`를 사용한다.
UI 컴포넌트는 `disabled`를 사용한다.
```

---

## 1. 도메인 엔티티는 `active`를 사용한다

### Decision

도메인에서 엔티티의 사용 가능 여부는 `active: boolean` 필드로 표현한다.

### Example

```ts
type Category = {
  active: boolean;
};
```

### Meaning

- `true`는 해당 엔티티가 새로운 작업에 유효함을 의미한다
- `false`는 엔티티가 시스템에는 남아 있지만, 더 이상 새로운 입력이나 선택에 사용할 수 없음을 의미한다

### `active`가 `false`가 되는 경우

일반적인 사례는 다음과 같다.

- 관리자가 카테고리를 폐기한 경우
- 기능 또는 서비스가 더 이상 지원되지 않는 경우
- 설정이 새로운 버전으로 대체된 경우

이런 경우 엔티티는 삭제되지 않으므로, 이력과 감사 가능성을 유지할 수 있다.

### Reason

- `active`는 비즈니스 의미를 명확하게 표현한다
- 엔티티 라이프사이클 의미와 잘 맞는다
- UI 구현 세부사항과 독립적으로 유지된다

---

## 2. UI 컴포넌트는 `disabled`를 사용한다

### Decision

UI에서 상호작용 상태는 `disabled`로 표현한다.

### Example

```tsx
<Select disabled={!category.active} />
```

### Meaning

- `disabled`는 UI 컨트롤이 상호작용 가능한지를 나타낸다
- 이 값은 도메인 상태, 권한, 로딩 상태, 혹은 그 밖의 표현 계층 조건으로부터 파생된다

### Reason

- `disabled`는 표준적인 UI 용어다
- HTML 의미 체계와 일치한다
- 프레젠테이션 계층을 명시적으로 유지한다

---

## 3. 분리 규칙

### Rule

```txt
`active`는 비즈니스 유효성을 설명한다.
`disabled`는 UI 상호작용 가능 여부를 설명한다.
```

### Practical Interpretation

- 도메인 모델에는 UI 용어가 포함되면 안 된다
- UI 컴포넌트는 도메인 상태로부터 `disabled`를 파생할 수 있다
- 반대로 UI 상태가 도메인 모델을 정의해서는 안 된다

---

## 4. Soft Delete와의 관계

### Interpretation

`active` 플래그는 소프트 삭제 메커니즘처럼 동작할 수 있다.

- 엔티티는 물리적으로 삭제되지 않는다
- `active = false`는 논리적 비활성화를 의미한다
- 이력 데이터는 감사와 리포팅을 위해 그대로 유지된다

### Benefits

이 접근은 다음을 지원한다.

- 감사 가능성
- 데이터 무결성
- 일관된 리포팅

---

## Rationale

### 도메인 모델 전반의 일관성

시스템은 엔티티 수준 상태에 대해 일관되게 `active`를 사용한다.

- `Category`
- `Ticket`
- `Attachment`

이 패턴을 유지하면 가독성이 높아지고, 공유된 설계 규칙이 더 분명해진다.

---

### 관심사의 명확한 분리

- `active`는 비즈니스/도메인의 책임이다
- `disabled`는 UI/프레젠테이션의 책임이다

이 분리는 아키텍처 경계를 깔끔하게 유지한다.

---

### 의미 누수 감소

도메인 모델에서 `disabled`를 사용하면 백엔드와 공유 타입에 UI 의미가 스며들게 된다.

`active`를 선택하면 모델은 다음 전반에서 이식 가능하게 유지된다.

- 도메인 로직
- API 계약
- 문서화
- 프론트엔드 렌더링

---

## Alternatives Considered

### 1. 도메인에서 `disabled` 사용

기각.

이유:

- UI 의미와 너무 밀접하게 연결되어 있다
- 비즈니스 레벨에서 모호하다
- 모델과 프레젠테이션의 분리를 약화시킨다

---

### 2. `enabled` 사용

기각.

이유:

- 라이프사이클 의미로는 덜 자연스럽다
- 상태라기보다 토글처럼 읽힌다
- 기존 도메인 용어와 일관되지 않는다

---

### 3. `deactive` 사용

기각.

이유:

- 표준 영어 표현이 아니다
- 코드와 문서에서 혼란을 만들 가능성이 높다

---

## Implementation Guideline

- 도메인 엔티티에는 항상 `active`를 저장한다
- API 응답에서 도메인 상태 필드로 `disabled`를 노출하지 않는다
- UI 상호작용 상태는 항상 도메인 상태로부터 파생한다

### Example

```ts
const disabled = !entity.active;
```

---

## Impact

### 도메인 모델링에 대한 영향

- 엔티티는 작업상 유효성을 표현하기 위해 `active`를 사용한다
- 비활성화된 엔티티도 이력 참조를 위해 계속 남아 있다

---

### UI 구현에 대한 영향

- 컴포넌트는 도메인 규칙과 UI 조건으로부터 `disabled`를 파생한다
- UI 상태는 명시적이고 조합 가능하게 유지된다

---

### 네이밍 일관성에 대한 영향

- 같은 개념이 시스템 전반에서 일관되게 표현된다
- 리뷰어는 도메인 로직과 UI 동작을 빠르게 구분할 수 있다

---

## Key Insight

```txt
도메인 언어는 비즈니스 의미를 설명해야 하고,
UI 언어는 상호작용 동작을 설명해야 한다.
```

---

## Summary

이 프로젝트는 다음과 같은 명확한 네이밍 규칙을 채택한다.

- 엔티티 수준의 비즈니스 상태에는 `active`를 사용한다
- UI 상호작용 상태에는 `disabled`를 사용한다

이를 통해 다음을 유지할 수 있다.

- 깔끔한 도메인 모델
- 명시적인 UI 의미 체계
- 일관되고 확장 가능한 전체 시스템
