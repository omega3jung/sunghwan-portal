# 2025-12 Naming Decision

## Context

Service Desk 프로젝트를 설계하면서 **UI 컴포넌트, form mode, i18n key, component API** 전반에 걸쳐
일관된 naming 전략이 필요했다.

특히 다음과 같은 상황에서 naming ambiguity가 발생했다:

- Settings vs Preferences
- create vs edit vs view mode
- field translation key 구조
- helper function vs explicit usage
- component API (children vs trigger)

초기에는 단순한 naming으로 시작했지만,
점차 **“이름이 곧 설계”**라는 문제를 인식하게 되었다.

---

## Problem

### 1. Semantic Ambiguity

#### Example: Settings

“Settings”라는 용어는 일반적으로:

- System-level configuration
- Admin configuration

을 의미하는 경우가 많다.

하지만 실제 구현하려는 기능은:

- 언어 변경
- 테마 변경 (light/dark)
- 컬러 테마

👉 즉, **User Preference 영역**

---

### 2. Mode Naming Ambiguity

초기 form props:

```ts
type?: "create" | "edit";
```

문제:

- create인데 edit 모드로 동작하는 구조
- 의미적으로 어색함
- 내부 로직 (POST vs PUT)과 분리되지 않음

---

### 3. i18n Naming & Access

초기 고민:

```ts
t(`field.${name}.label`);
```

vs

```ts
fieldLabel(name);
```

문제:

- abstraction vs explicitness
- 협업 시 가독성 문제
- HR/비개발자도 이해 가능한 구조 필요

---

### 4. Component API Naming

Dialog trigger 설계:

```tsx
<Dialog trigger={...} />
```

vs

```tsx
<Dialog>{trigger}</Dialog>
```

문제:

- flexibility vs explicitness
- API 일관성

---

## Decision Summary

```txt id="naming-core"
이름은 구현이 아니라 “의미”를 기준으로 정의한다.
```

---

## 1. Settings → Preferences

### Decision

```txt id="pref-decision"
User-facing 설정은 Settings가 아니라 Preferences로 명명한다.
```

### Rationale

- Settings = system / admin configuration
- Preferences = user personalization

### Result

```tsx
<PreferencesMenu />
```

👉 사용자 중심 UX semantics 확보

---

## 2. Form Mode Naming

### Decision

```ts
type Mode = "create" | "update" | "view";
```

### Rationale

- edit → 모호함
- update → API semantics와 일치 (PUT)
- create/update는 필드가 같아도 **의미가 다름**

👉 와 일관성 유지

---

### Key Insight

```txt id="mode-insight"
같은 UI라도, 다른 lifecycle이면 다른 이름을 사용한다.
```

---

## 3. Explicit i18n Key Strategy

### Decision

```txt id="i18n-decision"
Helper abstraction 대신 explicit key 사용
```

### Example

```ts
t("field.title.label", { ns: "common" });
```

### Rationale

- 코드만 봐도 의미 파악 가능
- 비개발자도 이해 가능
- hidden logic 제거

👉 반영

---

### Rejected Option

```ts
fieldLabel("title");
```

❌ 문제:

- 어디 namespace인지 불명확
- abstraction으로 인해 추적 어려움

---

## 4. Field Naming Structure

### Decision

```json
{
  "field": {
    "title": {
      "label": "...",
      "placeholder": "..."
    }
  }
}
```

### Rationale

- label/placeholder 통합 관리
- 재사용성 증가
- 구조적 consistency 확보

---

## 5. Component API Naming

### Decision

```txt id="api-decision"
trigger prop을 기본으로 사용하되,
필요 시 children 패턴을 허용한다.
```

### Example

```tsx
<Dialog trigger={<Button />} />
```

or

```tsx
<Dialog>{trigger}</Dialog>
```

---

### Rationale

- 명시적 API 유지
- 유연성 확보

---

### Insight

```txt id="api-insight"
컴포넌트 API는 “사용 방식”이 드러나야 한다.
```

---

## 6. Naming Principle

### Rule 1

```txt id="rule-1"
이름은 역할이 아니라 의미를 표현해야 한다
```

---

### Rule 2

```txt id="rule-2"
UI가 같아도 lifecycle이 다르면 이름을 분리한다
```

---

### Rule 3

```txt id="rule-3"
추상화보다 명시성을 우선한다
```

---

### Rule 4

```txt id="rule-4"
비개발자도 이해 가능한 naming을 지향한다
```

---

## Trade-offs

### Pros

- 의미 기반 설계 가능
- 코드 가독성 향상
- 협업 용이
- domain 이해도 상승

---

### Cons

- 다소 verbose한 naming
- 초기 설계 시간 증가

---

## Impact

### Code Quality

- naming만 보고도 구조 이해 가능

---

### Documentation

- 코드와 문서 간 일관성 확보

---

### Collaboration

- HR / 기획자 / 디자이너도 이해 가능

---

## Final Insight

```txt id="final-insight"
Naming은 구현 디테일이 아니라
시스템 설계를 드러내는 가장 작은 단위이다.
```

이 결정은 단순 naming 정리가 아니라,
**도메인 중심 설계를 코드 레벨까지 일관되게 반영하기 위한 선택**이다.
