# README 전략

## 목표

이 문서는 프로젝트 전반에서 README 파일을 어떻게 구성하고 배치할지 정의하며,
다음 목표를 지원한다.

- 리뷰어에게 명확한 첫인상을 제공
- 다국어 접근성을 확보
- 확장 가능한 문서 구조를 유지
- 저장소 진입점과 심화 문서를 분리

---

## 핵심 원칙

```txt
README는 전체 문서가 아니라 진입점이다
```

루트 README는 독자를 안내해야 하며, 과도한 정보로 부담을 주어서는 안 된다.

---

## 구조 개요

### 루트 레벨

```txt
/README.md
/README.ko.md
/docs/
```

---

## 언어 전략

### 접근 방식

각 언어는 별도의 README 파일을 가진다.

- `README.md` -> 영어(기본)
- `README.ko.md` -> 한국어

---

### 언어 전환 섹션

루트 README 상단에는 다음과 같은 언어 전환 섹션을 둔다.

```md
## **Languages**

- [English](../../../README.md)
- [Korean](../../../README.ko.md)
```

---

### 근거

- GitHub는 기본적으로 `README.md`를 표시한다.
- 언어를 섞지 않아도 각 문서를 끝까지 자연스럽게 읽을 수 있다.
- 인라인 번역 구조의 복잡성을 피할 수 있다.

---

## 루트 README의 역할

루트 README는 다음 목적을 가진다.

### 1. 프로젝트를 빠르게 설명한다

- 이 프로젝트가 무엇인지
- 왜 존재하는지
- 어떤 점이 흥미로운지

---

### 2. 핵심 설계 주제를 강조한다

예시:

- 도메인 주도 설계
- 기능 기반 아키텍처
- SLA / 승인 / 할당 시스템

---

### 3. 더 깊은 문서로 이동할 수 있게 안내한다

```md
See more in: docs/en/README.md
```

---

### 4. 기술적 신뢰감을 보여준다

- 기술 스택
- 프로젝트 구조
- 핵심 아키텍처 결정

---

## 루트 README에 넣지 말아야 할 것

다음 내용은 피한다.

- 깊은 도메인 설명
- 전체 아키텍처의 상세 분해
- 긴 의사결정 배경 설명

이런 내용은 `docs/`에 둔다.

---

## README와 Docs의 관계

### 원칙

```txt
README -> 개요
Docs -> 시스템 설명
```

---

### 흐름

```txt
README -> docs/en/README.md -> 상세 문서들
```

---

## Docs 구조

현재 구조:

```txt
docs/
  en/
    01-project/
    02-architecture/
    03-domain/
    04-ui-ux/
    05-data-fetching/
    06-form-design/
    07-i18n/
    08-dev-strategy/
```

---

### 왜 Docs를 분리하는가?

- README를 가볍게 유지할 수 있다.
- 문서를 확장 가능한 구조로 운영할 수 있다.
- 도메인 중심 설명이 가능해진다.

---

## 이 전략이 필요한 이유

### 1. 리뷰어 경험

리뷰어, 특히 채용 과정의 평가자는 보통 다음 흐름으로 저장소를 본다.

1. 저장소를 연다
2. README를 읽는다
3. 더 볼지 말지 결정한다

명확한 README는 이 흐름을 더 좋게 만든다.

---

### 2. 관심사의 분리

- README -> 커뮤니케이션 레이어
- docs -> 시스템 설계 레이어

---

### 3. 확장성

프로젝트가 커져도:

- README는 안정적으로 유지된다.
- docs는 자유롭게 확장할 수 있다.

---

### 4. 유지보수성

- 긴 설명을 중복해서 적지 않아도 된다.
- 섹션별 업데이트가 쉬워진다.

---

## 고려한 대안

### 1. 모든 내용을 하나의 README에 담는 방식

- 장점: 단순하다
- 단점: 너무 길어진다
- 단점: 탐색이 어렵다

---

### 2. 인라인 다국어 README

- 장점: 파일이 하나다
- 단점: 가독성이 떨어진다
- 단점: 유지보수가 어렵다

---

### 3. 별도 `readme/` 폴더 사용

```txt
/readme/README.ko.md
```

- 장점: 정리된 느낌을 줄 수 있다
- 단점: GitHub가 자동으로 렌더링하지 않는다
- 단점: 첫인상이 약해진다

---

## 결정

- README 파일은 루트 레벨에 둔다
- 언어별 파일(`README.md`, `README.ko.md`)을 사용한다
- 상세 설명은 `docs/`로 이동한다

---

## 설계 원칙과의 정렬

이 전략은 다음 원칙과 맞닿아 있다.

- 완전성보다 명확성
- 관심사의 분리
- 리뷰어 우선 설계
- 확장 가능한 문서 아키텍처

---

## 요약

README 전략은 다음을 보장한다.

- 프로젝트가 강한 첫인상을 남긴다
- 문서 구조가 계속 확장 가능하다
- 다국어 지원이 깔끔하고 유지보수 가능하다
- 시스템 설계와 프로젝트 소개가 적절히 분리된다

그 결과 저장소는 **실제로 동작하는 프로젝트**이면서 동시에
**잘 문서화된 포트폴리오 산출물**로 기능할 수 있다.
